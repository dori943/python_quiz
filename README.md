안녕하세요, 루키마리너 김도희입니다.

평가를 하다보니 생긴 불편한 점들이 있어 제안하고자 말씀 올립니다!!
평가요청시 수락 후에 100포인트 차감되었으면 좋겠으며, 동료평가자로 지정될 시 수락 후 평가자 불참 버튼을 누를 수 있도록 해주세요.

평가요청 시 바로 -100포인트가 차감되고, 동료평가자로 잡히신 분이 수락을 하지 않으시고 시간이 지나도 수락을 안하셨을 경우에는 제가 요청취소를 해야하는데 그렇게 되면 제가 이미 차감된 점수를 받을 수 없게 됩니다.
공부하고 있는 와중에 알림을 못 보는 경우도 많으신 것 같은데, 평가자에게 수락을 받은 다음에 피평가자의 점수가 -100 차감되는게 어떨까요?? 아무것도 하지 못하고 의미없이 포인트만 깎이는 것 같습니다.

또한, 동료평가자로 지정되었을 시에도 현재 시각이 예를 들면 12시라고 했을 때, 평가가 12시 7분 이렇게 나중에 잡히기도 하는데 그럴 경우 30분 안에 수락을 하는 것보다 시간이 단축되잖아요. 이 부분이 허점인 것 같습니다. 그래서 연락 온지 모른체 있으면 피평가자로부터 금방 평가취소가 되고 사유는 평가자 불참으로 되어 포인트가 -50 차감됩니다.
평가요청이 와서 바로 확인하지 못한 제 잘못도 있지만, 수락도 하지 않은채 약속을 지키지 않은 것도 아닌데도 포인트가 차감되는 것은 좀 아까운 것 같습니다.

# 🎯 파이썬 퀴즈 게임

## 프로젝트 개요

파이썬 프로그래밍 지식을 퀴즈 형식으로 테스트할 수 있는 콘솔 기반 퀴즈 게임입니다.  
퀴즈 풀기, 추가, 삭제, 목록 확인, 점수 기록 기능을 제공하며, 게임 데이터는 `state.json` 파일에 저장되어 프로그램을 재실행해도 유지됩니다.

**주제: 파이썬 프로그래밍**

파이썬은 이 프로젝트 자체를 구현하는 언어이기도 하며, 이 미션의 핵심인 '파이썬 프로그래밍의 기초 배우기'를 공부할 수 있는 방법이라고 생각되어 이 주제로 선정하게 되었습니다.
퀴즈를 풀면서 파이썬 문법과 개념을 자연스럽게 복습할 수 있어 학습 효과와 게임의 재미를 동시에 얻을 수 있다고 판단해 선정했습니다.

### 추가 세부 기능

- **랜덤 출제**: 퀴즈 풀기 시 `random.sample()`로 문제 순서를 섞어 출제
- **문제 수 선택**: 전체 퀴즈 중 원하는 수만큼 선택해서 풀기 가능
- **힌트 기능**: 풀이 중 `H` 입력 시 힌트 표시, 해당 문제 점수 0.5점으로 차감
- **점수 기록 히스토리**: 게임 종료 시 날짜 / 문제 수 / 점수를 `state.json`에 누적 저장
- **예외 처리**: 빈 입력, 문자 입력, 범위 초과, `Ctrl+C`, `EOFError` 모두 처리
- **파일 복구**: `state.json`이 없거나 손상된 경우 기본 퀴즈 데이터로 자동 초기화

## 클래스 설계 및 아키텍처
 
### 함수 대신 클래스를 사용한 이유
 
단순 함수로 구현할 경우, 퀴즈 목록·최고 점수·게임 기록 같은 상태를 전역 변수로 관리해야 합니다.  
전역 변수는 어디서든 변경 가능하기 때문에 버그 추적이 어렵고, 코드가 커질수록 관리가 힘들어집니다.
 
클래스를 사용하면 상태(`self.quizzes`, `self.best_score` 등)와 그 상태를 다루는 기능(`play_quiz`, `save_data` 등)을 하나의 객체 안에 묶을 수 있어 데이터 흐름이 명확해지고 유지보수가 쉬워집니다.
 
### 클래스별 책임 분리
 
| 클래스 | 파일 | 책임 |
|--------|------|------|
| `Quiz` | `quiz.py` | 퀴즈 1개의 데이터(문제/선택지/정답/힌트) 표현, 출력, 정답 확인, 직렬화 |
| `QuizGame` | `quiz_game.py` | 게임 전체 흐름 관리, 메뉴 처리, 파일 입출력, 점수 계산 |
 
`Quiz`는 "퀴즈 하나가 무엇인지"만 알고, `QuizGame`은 "게임이 어떻게 돌아가는지"만 압니다.  
두 클래스가 서로의 내부 구현에 의존하지 않으므로, 한 클래스를 수정해도 다른 클래스에 영향이 최소화됩니다.
 
### 로직 분리 기준
 
메서드는 **하나의 메서드가 하나의 역할만 수행**하는 것을 원칙으로 분리했습니다.
 
| 메서드 | 역할 |
|--------|------|
| `show_menu()` | 메뉴 출력만 담당 |
| `get_menu_choice()` | 입력 검증만 담당 |
| `play_quiz()` | 게임 진행 로직만 담당 |
| `save_data()` / `load_data()` | 파일 입출력만 담당 |
 
입력 처리, 게임 로직, 저장 로직을 분리함으로써 특정 기능에 버그가 생겼을 때 해당 메서드만 수정하면 됩니다.
 
### 요구사항 변경 시 수정 위치
 
| 변경 상황 | 수정할 파일 / 클래스 |
|-----------|----------------------|
| 퀴즈에 난이도 속성 추가 | `quiz.py` - `Quiz` 클래스 |
| 기본 퀴즈 내용 변경 | `default_quizzes.py` |
| 메뉴 항목 추가/제거 | `quiz_game.py` - `show_menu`, `get_menu_choice`, `run` |
| 저장 파일 형식 변경 | `quiz_game.py` - `save_data`, `load_data` |
| 프로그램 시작점 변경 | `main.py` |

## 데이터 저장 설계
 
### JSON 파일을 선택한 이유
 
| 항목 | 설명 |
|------|------|
| 가독성 | 텍스트 기반이라 파일을 직접 열어 내용을 확인하고 수정 가능 |
| 호환성 | 파이썬 표준 라이브러리 `json` 모듈만으로 처리 가능, 별도 설치 불필요 |
| 구조 표현 | 퀴즈 목록(배열), 게임 기록(중첩 객체) 등 계층 구조를 자연스럽게 표현 가능 |
| 범용성 | 추후 웹 API나 다른 언어와 연동 시에도 그대로 사용 가능한 표준 포맷 |
 
### 파일 입출력 시 try/except가 필요한 이유
 
파일을 다룰 때는 코드가 올바르더라도 **외부 환경**에 의해 다양한 오류가 발생할 수 있습니다.
 
| 실패 케이스 | 발생 예외 | 처리 방법 |
|-------------|-----------|-----------|
| `state.json` 파일이 아직 없음 | `FileNotFoundError` | `os.path.exists()`로 사전 확인 |
| 파일이 깨졌거나 JSON 형식이 아님 | `json.JSONDecodeError` | 기본 퀴즈 데이터로 초기화 |
| 디스크 용량 부족, 파일 권한 없음 | `IOError`, `OSError` | 오류 메시지 출력 후 계속 진행 |
| JSON 키 누락 또는 타입 불일치 | `KeyError`, `TypeError` | 기본값으로 대체 |
 
try/except 없이 이런 상황이 발생하면 프로그램이 즉시 종료됩니다.  
예외를 잡아 처리함으로써 데이터가 없거나 손상된 환경에서도 프로그램이 안전하게 동작합니다.
 
### 현재 스키마를 이렇게 설계한 이유
 
```json
{
  "quizzes": [...],
  "best_score": 4.5,
  "has_played": false,
  "score_history": [...]
}
```
 
- **`quizzes`를 배열로 설계한 이유**: 퀴즈는 순서가 있고 반복 처리가 필요하므로 리스트 구조가 적합합니다.
- **`best_score`와 `has_played`를 분리한 이유**: `best_score`가 0점일 때 퀴즈를 한 번도 안 푼 상태와 구분이 불가능하므로, 별도 boolean flag로 명확하게 관리합니다.
- **`score_history`를 별도 배열로 둔 이유**: 최고 점수만 저장하면 과거 기록이 사라지므로, 모든 게임 기록을 누적 저장해 히스토리 조회가 가능하도록 설계했습니다.
 
### JSON 저장 방식의 한계
 
현재 구조는 소규모 데이터에 적합하지만, 데이터가 늘어날 경우 다음과 같은 한계가 있습니다.
 
| 한계 | 설명 |
|------|------|
| 전체 덮어쓰기 | 저장 시 파일 전체를 매번 다시 씁니다. 퀴즈가 수천 개가 되면 성능이 저하됩니다. |
| 동시 접근 불가 | 두 프로세스가 동시에 파일을 쓰면 데이터가 손상될 수 있습니다. |
| 검색 기능 없음 | 특정 조건의 퀴즈를 찾으려면 전체를 읽어 직접 순회해야 합니다. |
| 확장성 한계 | 데이터가 대용량이 되면 SQLite 등의 경량 데이터베이스로 전환이 필요합니다. |

## 데이터 파일 설명

### 경로

프로젝트 루트 디렉토리의 `state.json`

### 역할

- 등록된 퀴즈 목록, 최고 점수, 전체 게임 기록을 영구 저장
- 프로그램 시작 시 자동으로 불러오고, 퀴즈 추가/삭제/게임 종료 시 자동으로 저장
- 파일이 없으면 `default_quizzes.py`의 기본 퀴즈 5개로 초기화
- 파일이 손상된 경우 안내 메시지 출력 후 기본 데이터로 복구

### 스키마

```json
{
  "quizzes": [
    {
      "question": "문제 텍스트",
      "choices": ["선택지1", "선택지2", "선택지3", "선택지4"],
      "answer": 2,
      "hint": "힌트 텍스트"
    }
  ],
  "best_score": 4.5,
  "score_history": [
    {
      "date": "2025-01-01 14:30",
      "total": 5,
      "score": 4.5
    }
  ]
}
```

### 필드 설명

| 필드 | 타입 | 설명 |
|------|------|------|
| `quizzes` | array | 등록된 퀴즈 목록 |
| `quizzes[].question` | string | 문제 텍스트 |
| `quizzes[].choices` | array | 선택지 4개 |
| `quizzes[].answer` | integer | 정답 번호 (1~4) |
| `quizzes[].hint` | string | 힌트 (없으면 빈 문자열) |
| `best_score` | number / null | 최고 점수 (게임 전이면 `null`) |
| `has_played` | boolean | 퀴즈를 한 번이라도 풀었는지 여부 |
| `score_history` | array | 전체 게임 기록 목록 |
| `score_history[].date` | string | 게임 일시 (`YYYY-MM-DD HH:MM`) |
| `score_history[].total` | integer | 해당 게임에서 푼 문제 수 |
| `score_history[].score` | number | 해당 게임 점수 (힌트 사용 시 0.5점 단위) |

## Git 관리 기록
 
### 커밋 단위 분리 원칙
 
커밋은 **하나의 기능 완성 단위**로 분리했습니다.  
하나의 커밋에 여러 기능을 섞으면 특정 기능에 문제가 생겼을 때 되돌리기 어렵고 변경 이력을 파악하기 힘들기 때문입니다.  
커밋 메시지는 `태그: 내용` 형식으로 작성해 무엇을 했는지 한눈에 파악할 수 있도록 했습니다.
 
| 태그 | 의미 |
|------|------|
| `init` | 프로젝트 초기 설정 |
| `feat` | 새 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 수정 |
| `refactor` | 기능 변경 없이 코드 구조 개선 |

### 브랜치 생성 및 병합 기록
 
브랜치는 **main 브랜치를 항상 동작하는 상태로 유지**하고, 새 기능은 별도 브랜치에서 개발한 뒤 완성되면 main에 병합하는 방식으로 관리했습니다.  
이렇게 하면 개발 중인 불완전한 코드가 main에 섞이지 않아, 언제든 main 브랜치를 안전하게 실행할 수 있습니다.
 
```bash
# 퀴즈 풀기 기능을 별도 브랜치에서 개발
git checkout -b feature/play-quiz
 
# 기능 개발 및 커밋
git add .
git commit -m "feat: 퀴즈 풀기 기능 구현"
 
# main 브랜치로 돌아와 병합
git checkout main
git merge feature/play-quiz
 
# 병합 완료 후 브랜치 삭제
git branch -d feature/play-quiz
```
 
브랜치를 분리하는 이유는 기능 개발 도중 main을 건드리지 않기 위해서입니다.  
예를 들어 `feature/play-quiz` 작업 중에도 main은 항상 실행 가능한 상태를 유지하므로, 다른 기능을 긴급히 수정하거나 배포해야 할 때 main으로 바로 전환할 수 있습니다.
 
### clone 및 pull 사용 기록
 
**clone**: 최초 원격 저장소를 로컬로 복제할 때 사용했습니다.
 
```bash
git clone https://github.com/{username}/{repository}.git
```
 
**pull**: 원격 저장소의 변경 사항을 로컬에 반영할 때 사용했습니다.  
브랜치 병합 후 원격에 push된 내용을 로컬 main 브랜치에 동기화하는 용도로 사용했습니다.
 
```bash
git checkout main
git pull origin main
```
